The `@duckflux/runner` package ships a `quack` binary that lets you run, lint, and validate workflows from the command line.

```bash
quack <command> <workflow.yaml> [flags]
```

<Tabs>
  <TabItem label="Universal">
    Auto-detects your package manager (apt, brew, bun, npm) and installs accordingly. Falls back to downloading a standalone binary.

  </TabItem>

  <TabItem label="macOS (brew)">

    ```bash

    ```
  </TabItem>

  <TabItem label="Linux (apt)">
    Install via apt:

    ```bash
    curl -fsSL https://duckflux.github.io/apt-repo/install.sh | bash
    ```
  </TabItem>

  <TabItem label="npm/bun">
    Install globally so `quack` is available everywhere on your machine.

    ```bash
    # npm
    npm install -g duckflux

    # Bun
    bun add -g duckflux
    ```

    Verify:

    ```bash
    quack version
    ```
  </TabItem>

  <TabItem label="npx/bunx">
    Run without installing — always uses the latest published version.

    ```bash
    # Node.js
    npx duckflux run workflow.yaml

    # Bun
    bunx duckflux run workflow.yaml
    ```
  </TabItem>

  <TabItem label="Binaries">

    | macOS ARM64 (Apple Silicon) | `quack-darwin-arm64` |

    | Linux x64 | `quack-linux-x64` |
    | Linux ARM64 | `quack-linux-arm64` |
    | Windows x64 | `quack-windows-x64.exe` |

    **macOS / Linux:**

    ```bash
    # Example for macOS ARM64 — replace the filename for your platform
    curl -fsSL https://github.com/duckflux/runtime-js/releases/latest/download/quack-darwin-arm64 \
      -o /usr/local/bin/quack
    chmod +x /usr/local/bin/quack
    quack version
    ```

    **Windows:** download the `.exe`, place it somewhere on your `%PATH%`, and run it directly.
  </TabItem>

| Command | Description |
|---------|-------------|
| [`run`](#run) | Execute a workflow |
| [`lint`](#lint) | Validate schema and detect common issues (no execution) |
| [`validate`](#validate) | Full validation including input schema |
| [`server`](#server) | Start the web server UI for visual workflow observation |
| [`version`](#version) | Print the CLI version |

## `run`

Executes a workflow from a YAML file and prints the result to `stdout`.

```bash
quack run <workflow.yaml> [flags]
```

`run` is the default command — if the first positional argument is a `.yaml` file and no subcommand is given, it is implied:

```bash
quack workflow.yaml   # same as: quack run workflow.yaml
```

### Flags

| Flag | Short | Type | Default | Description |
|------|-------|------|---------|-------------|
| `--input` | `-i` | string | — | Input as `key=value`. Repeatable. Values are JSON-parsed, falling back to strings. |
| `--input-file` | — | string | — | Path to a JSON file containing workflow inputs. |
| `--cwd` | — | string | — | Working directory for `exec` participants. See [Working directory](/runtime/working-directory/). |
| `--event-backend` | — | string | `memory` | Event hub backend: `memory`, `nats`, or `redis`. |
| `--nats-url` | — | string | — | NATS server URL. Required when `--event-backend=nats`. |
| `--nats-stream` | — | string | `duckflux-events` | JetStream stream name. |
| `--redis-addr` | — | string | `localhost:6379` | Redis server address (`host:port`). |
| `--redis-db` | — | string | `0` | Redis database number. |
| `--trace-dir` | — | string | — | Directory to write execution trace files. One file per run, named `<executionId>.<ext>`. |
| `--trace-format` | — | string | `json` | Trace output format: `json`, `txt`, or `sqlite`. |
| `--verbose` | `-v` | boolean | `false` | Print error stack traces to `stderr`. |
| `--quiet` | — | boolean | `false` | Suppress `stdout` output. |

### Input sources

Inputs are merged from multiple sources. Higher entries take priority:

1. `--input` flags (highest)
2. `--input-file` contents
3. JSON piped via `stdin` (when no other input source is provided)

### `--input` value parsing

Values are JSON-parsed first, falling back to plain strings:

```bash
--input name=World        # {name: "World"}
--input count=3           # {count: 3}
--input active=true       # {active: true}
--input data='{"x":1}'    # {data: {x: 1}}
--input items='[1,2,3]'   # {items: [1, 2, 3]}
--input flag              # {flag: true}  (no "=" → boolean true)
```

### Output

- `string` output → printed as-is (no trailing newline added)
- Object/array output → printed as formatted JSON (2-space indent)
- `null` / `undefined` output → nothing printed

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | Workflow succeeded |
| `1` | Execution error (file not found, parse error, unhandled exception) |
| `2` | Workflow ran but returned `success: false` |

### Examples

```bash
# Basic execution
quack run workflow.yaml

# Pass inputs
quack run workflow.yaml --input name=World --input count=3

# Inputs from file
quack run workflow.yaml --input-file inputs.json

# Piped inputs
cat inputs.json | quack run workflow.yaml

# Custom working directory with verbose errors
quack run workflow.yaml --cwd /workspace --verbose

# NATS event backend
quack run workflow.yaml \
  --event-backend=nats \
  --nats-url=nats://localhost:4222

# Redis event backend
quack run workflow.yaml \
  --event-backend=redis \
  --redis-addr=localhost:6379

# Write a JSON trace for each run
quack run workflow.yaml --trace-dir ./traces

# Write a SQLite trace (all runs in separate DB files)
quack run workflow.yaml --trace-dir ./traces --trace-format sqlite
```

### Tracing

Use `--trace-dir` to capture a structured execution record for each run. See [Execution tracing](/runtime/tracing/) for full details.

## `lint`

Validates a workflow file without executing it. Checks schema correctness, semantic consistency, and common authoring issues.

```bash
quack lint <workflow.yaml>
```

### Checks performed

1. **Schema validation** — the file must conform to the duckflux JSON schema.
2. **Semantic validation** — participant references, variable bindings, and imports are verified.
3. **Lint warnings** (non-blocking) — the following patterns are flagged:
   - Loops without both `until` and `max` conditions (infinite-loop risk)
   - Parallel branches writing to the same context variable (race condition risk)
   - Inline participants without an `as` name (output unreachable by name)

If all checks pass, `valid` is printed to `stdout`. Errors and warnings go to `stderr`.

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | Workflow is valid |
| `1` | Validation failed or file could not be read |

### Example

```bash
quack lint workflow.yaml

# CI-friendly: capture all output
quack lint workflow.yaml 2>&1

# Check exit code
if quack lint workflow.yaml; then
  echo "Ready to run"
fi
```

## `validate`

Performs the same checks as [`lint`](#lint), and additionally validates provided inputs against the workflow's declared `inputs` schema.

```bash
quack validate <workflow.yaml> [flags]
```

### Flags

| Flag | Short | Type | Default | Description |
|------|-------|------|---------|-------------|
| `--input` | `-i` | string | — | Input as `key=value`. Repeatable. |
| `--input-file` | — | string | — | Path to a JSON file containing inputs. |

`stdin` is also accepted (same rules as `run`).

### Checks performed

1. Schema validation
2. Semantic validation
3. **Input schema validation** — types, `enum`, `min`, `max`, `pattern`, `format`, and required fields are all checked against the values you provide.

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | All validations passed |
| `1` | Validation failed |

### Examples

```bash
# Validate structure only
quack validate workflow.yaml

# Validate structure + inputs
quack validate workflow.yaml --input name=World --input count=3

# Inputs from file
quack validate workflow.yaml --input-file test-inputs.json

# Piped inputs
echo '{"name":"World"}' | quack validate workflow.yaml
```

## `server`

Starts the duckflux web server UI — a local dev tool for executing and observing workflows visually. See [Web Server UI](/tooling/web-server-ui/) for a full feature walkthrough with screenshots.

```bash
quack server [flags]
```

### Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--trace-dir` | string | — | **Required.** Directory to watch for execution trace JSON files. |
| `--workflow-dir` | string | cwd | Directory to scan for `.yaml` / `.yml` workflow files. |
| `--port` | string | `3000` | HTTP port for the web UI. |

### Auto-install

If `@duckflux/server` is not installed in the current project, the CLI prompts to install it automatically:

```
@duckflux/server is not installed. Install it now? [Y/n]
```

Answering **Y** runs `bun add @duckflux/server -D` (falling back to `npm install` if Bun is not available).

### Examples

```bash
# Start with trace output directory
quack server --trace-dir ./traces

# Custom workflow directory and port
quack server --trace-dir ./traces --workflow-dir ./workflows --port 8080
```

<Aside type="tip">
  The server watches `--trace-dir` for new or updated trace files in real time. Run workflows via the UI's execute button or from the CLI in another terminal — both feed into the same trace directory.
</Aside>

## `version`

Prints the CLI version to `stdout`.

```bash
quack version
# 0.7.0
```

Always exits with code `0`.

## Event hub backends

The `--event-backend` flag (used with `run`) selects the pub/sub layer for [`emit`](/workflows/events/) and [`wait`](/workflows/events/#waiting-for-events) participants.

| Backend | Flag value | Infrastructure | Replay | Cross-process |
|---------|-----------|---------------|--------|--------------|
| In-memory | `memory` (default) | None | Yes | No |
| NATS JetStream | `nats` | NATS server | No | Yes |
| Redis Streams | `redis` | Redis server | Yes | Yes |

<Aside type="tip">
  The default `memory` backend requires no external services and supports in-process replay. Switch to `nats` or `redis` when you need cross-process event delivery.
</Aside>

For detailed configuration of each backend see [Event hub providers](/runtime/event-hub-providers/).

## Error handling

**Unknown command** — prints available commands and exits with code `1`.

**Missing file argument** — prints the command's usage string to `stderr` and exits with code `1`.

**Backend not installed** — if `@duckflux/hub-nats` or `@duckflux/hub-redis` is missing, the CLI prints an installation hint and exits with code `1`:

```
Error: NATS backend requires @duckflux/hub-nats — install it with: bun add @duckflux/hub-nats
```

**`--nats-url` missing** — required when `--event-backend=nats`; exits with code `1` if omitted.

**`--verbose` / `-v`** — appends the full stack trace to `stderr` for any error, regardless of command.

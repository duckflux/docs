Run any DuckFlux workflow with a single command. Or embed the engine directly in your TypeScript app. Either way, you get the same full-featured runtime — with no vendor lock-in, no agent framework to learn, and no infrastructure to manage by default.

```bash
quack run workflow.yaml --input env=production
```

---

## Two ways to use it

### CLI — zero setup

Install once, run anywhere. The `quack` binary reads your YAML, validates it, and executes it. No code, no build step, no config files.

```bash
# Run from npx/bunx without installing
bunx duckflux run workflow.yaml --input name=World

# Or install globally
bun add -g duckflux
quack run workflow.yaml
```

Works with [Bun](https://bun.sh) and Node.js. Inputs from flags, files, or stdin. Structured output to stdout. Exit codes for CI. See the [CLI reference](/runtime/cli/) for all flags.

### Library — embed in your app

Drop [`@duckflux/core`](https://www.npmjs.com/package/@duckflux/core) into any TypeScript project and call `executeWorkflow` directly. No subprocess, no glue scripts, no serialization overhead — the workflow runs in-process.

```typescript
import { executeWorkflow } from "@duckflux/core/engine";
import { parseWorkflowFile } from "@duckflux/core/parser";

const workflow = await parseWorkflowFile("./pipeline.yaml");
const result = await executeWorkflow(workflow, { env: "production" });

console.log(result.output);  // structured output
console.log(result.steps);   // per-step results, timings, errors
```

Full TypeScript types. Programmatic input/output. No side effects unless you ask for them. See the [library API reference](/runtime/library/).

## What you get out of the box

### Full spec coverage

The runtime implements the complete [DuckFlux v0.7 spec](https://github.com/duckflux/spec/blob/main/SPEC.md) — every participant type, every flow construct, every expression feature:

- **[Participants](/workflows/participant-types/)** — `exec` (shell), `http`, `emit` (events), `workflow` (sub-workflows)
- **Control flow** — [`loop`](/workflows/loops/), [`parallel`](/workflows/parallel/), [`if`/`else`](/workflows/conditionals/), `when` guards, [`set`](/workflows/set/), [`wait`](/workflows/events/)
- **[I/O chaining](/workflows/inputs-outputs/)** — step output flows automatically as input to the next step
- **[Expressions](/workflows/variables-and-expressions/)** — full [CEL](https://cel.dev) standard library with `has`, `size`, `matches`, `timestamp`, `duration`, and more
- **[Error strategies](/workflows/error-handling/)** — `fail`, `skip`, `retry` with exponential backoff, and fallback participants
- **Timeouts** — per-step or global, with override precedence

### Event hub included

Async workflows that emit and wait on events work out of the box with the built-in in-memory hub — no external services needed. Scale up to [NATS](https://nats.io) or [Redis](https://redis.io) when you need cross-process delivery.

| Backend | Extra package | Cross-process | Replay |
|---------|--------------|--------------|--------|
| In-memory | none | No | Yes |
| [NATS JetStream](https://docs.nats.io/nats-concepts/jetstream) | [`@duckflux/hub-nats`](https://www.npmjs.com/package/@duckflux/hub-nats) | Yes | No |
| [Redis Streams](https://redis.io/docs/latest/develop/data-types/streams/) | [`@duckflux/hub-redis`](https://www.npmjs.com/package/@duckflux/hub-redis) | Yes | Yes |

See [Event hub providers](/runtime/event-hub-providers/) for setup and configuration.

### Execution tracing <Badge text="New" variant="tip" />

Every run can produce a structured trace — one file per execution, written incrementally as each step completes. Choose the format that fits your workflow: JSON for downstream processing, TXT for quick human inspection, or [SQLite](https://sqlite.org) for SQL queries across multiple runs.

```bash
# Trace to JSON (default)
quack run workflow.yaml --trace-dir ./traces

# Query with any SQL client
quack run workflow.yaml --trace-dir ./traces --trace-format sqlite
```

Each trace captures every step — participants and control-flow constructs alike — with timing, inputs, outputs, errors, and retry counts. See [Execution tracing](/runtime/tracing/) for the full format reference.

## Packages

The runtime ships as focused, composable packages:

| Package | Install | Use when |
|---------|---------|----------|
| [`duckflux`](https://www.npmjs.com/package/duckflux) | `bun add -g duckflux` | You want the CLI |
| [`@duckflux/core`](https://www.npmjs.com/package/@duckflux/core) | `bun add @duckflux/core` | You want to embed the engine |
| [`@duckflux/hub-nats`](https://www.npmjs.com/package/@duckflux/hub-nats) | `bun add @duckflux/hub-nats` | You need [NATS](https://nats.io) event delivery |
| [`@duckflux/hub-redis`](https://www.npmjs.com/package/@duckflux/hub-redis) | `bun add @duckflux/hub-redis` | You need [Redis](https://redis.io) event delivery |

<Aside type="tip">
  `@duckflux/core` has zero mandatory runtime dependencies. Event hub packages are optional and only loaded when you use them.
</Aside>
